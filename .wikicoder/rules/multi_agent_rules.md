# 多 Agent 专家协作规范 (Multi-Agent Orchestration)

在处理复杂任务时，你可以像人类架构师一样，派生出专注于特定领域的“子专家”来协助你。

## 1. 适用场景 (When to spawn)
当你遇到以下情况时，建议使用 `spawn_subagent`：
- **深度分析**：需要对大量文件（如 50 个以上的日志或代码文件）进行地毯式扫描。
- **孤立任务**：任务逻辑与主线任务高度独立（如：编写一套完整的单元测试、生成一份详细的 API 文档）。
- **风险隔离**：需要执行一些可能失败但又不希望中断主进程逻辑的尝试性操作。

## 2. 任务描述准则 (The sub_task Prompt)
派生子 Agent 时，你的 `sub_task` 描述必须极其清晰，包含：
- **上下文背景**：简述项目当前的状态。
- **具体交付物**：明确要求子 Agent 产出什么（如：一个文件、一段分析报告或修复后的代码）。
- **约束条件**：告知子 Agent 必须遵守的特定规则。

## 3. 结果合并 (Result Consolidation)
- 子 Agent 结束后会返回一个最终报告。
- 你应当仔细阅读该报告，并将其中的关键成果合并到你的主任务进度中。
- 如果子 Agent 失败了，你必须在 `self_criticism` 中分析失败原因，并决定是自己接手还是重新派生新的专家。

> 记住：你是统筹者（Leader），子 Agent 是你的执行手。保持全局视角，不要陷入子任务的细节。
